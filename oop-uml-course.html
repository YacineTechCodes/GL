01 · Foundations

What is Modeling?

"A good sketch is worth more than a long speech." — Napoléon Bonaparte

Before writing a single line of code, software engineers need to think about and communicate the system they're building. That's where modeling comes in.

🗺️
Model
An abstract representation of reality — like a map represents a territory.
✏️
Modeling
Creating a model of a real system using languages and notations to simplify complexity.
🔭
Abstraction
Focusing on the important aspects and ignoring irrelevant details.
🏗️
Conception
A description of the future software application to be built.

Why Do We Model?

Modeling serves three key purposes in software development:

Purpose 1 — Simplification

Complex systems become manageable by excluding unnecessary details and focusing on what matters.

Purpose 2 — Shared Understanding

All stakeholders (clients, developers, managers) can look at the same model and understand the system the same way.

Purpose 3 — Communication

Models act as a universal language between different parties involved in a project.

For a model to work, it requires semantics and notation (how concepts look and what they mean) plus construction rules (how to correctly put them together).

02 · Foundations

Modeling Languages

A modeling language provides the vocabulary and grammar for building models. There are three major categories, each with different trade-offs.

Type Features Examples Trade-offs
Formal Mathematically verifiable, rigorous syntax Z language, VDM Precise but difficult to use
Semi-formal Easy, flexible, allows some ambiguity UML, Merise, pseudo-code Practical balance for most projects
Informal Expressive, no rules, natural language Plain text, diagrams on whiteboards Easy to express, hard to verify
Key Takeaway

In software engineering, semi-formal languages like UML are the sweet spot — precise enough to be useful, flexible enough to be practical. [Uschold, 1996]

Common Semi-Formal Methods

📐
UML
The industry standard for object-oriented software modeling. 14 diagram types.
🗃️
Merise
A French methodology popular for data-focused systems design using entity-relationship diagrams.
📝
Pseudo-code
Human-readable code-like notation for describing algorithms without syntax constraints.
03 · Foundations

Functional Decomposition

Before object-oriented approaches became popular, software was designed using functional decomposition — a top-down, hierarchical breakdown of what a program does.

🎯
Highlight Functions
The main goal is to identify and describe every function the system must perform.
⬇️
Top-Down
Start from the main program and break it into smaller and smaller sub-functions.
🔀
Data Separation
The problem logic is kept separate from data representation — a key limitation of this approach.
Example Method — SADT

Structured Analysis and Design Technique is a classic functional decomposition method that uses hierarchical diagrams to break a main program into a tree of sub-functions.

Limitation of Functional Decomposition

When the system's functions change, the entire hierarchy can collapse. Data and behavior are disconnected, making maintenance difficult. This led to the rise of the object-oriented approach.

04 · OOP

The Object-Oriented Paradigm

Object-oriented modeling shifts focus from what a system does to what a system is made of. Everything is modeled as an object that belongs to a class.

📦
Object
An instance with three aspects:
Identity (who it is) + State (its data) + Behavior (what it does)
🏛️
Class
A blueprint — a grouping of objects that share the same structure and behavior.

Example: A Class and Its Objects

Car
— brand : String
— speed : int
— color : String
+ accelerate()
+ brake()
+ getSpeed()
«class»
myCar : Car
brand = "Toyota"
speed = 80
color = "Red"
yourCar : Car
brand = "BMW"
speed = 120
color = "Blue"

Relationships Between Classes

Objects don't exist in isolation — they are connected through different types of relationships:

Association
A general relationship — "A uses B". E.g., a Student uses a Library.
Aggregation
A "has-a" relationship where parts can exist independently. E.g., a Team has Players.
Composition
A strong "made-of" relationship. Parts cannot exist without the whole. E.g., a House has Rooms.
Inheritance
A "is-a" relationship. E.g., an ElectricCar is-a Car — it inherits all Car properties.
05 · OOP

Core OOP Principles

The object-oriented paradigm is built on three foundational principles that make software more modular, reusable, and maintainable.

🔒
Encapsulation
Bundling data and methods together, hiding internal details. Objects expose only what's necessary through a public interface. Protects data integrity.
🧬
Inheritance
A child class inherits attributes and methods from a parent class. Promotes code reuse. Example: Dog inherits from Animal.
🎭
Polymorphism
The same method name can behave differently depending on the object. E.g., speak() says "Woof" for Dog and "Meow" for Cat.

Encapsulation in Code

class BankAccount { // Private — hidden from outside private balance: float // Public — accessible interface public deposit(amount: float) { if (amount > 0) balance += amount } public getBalance(): float { return balance } }

Inheritance in Code

class Animal { name: String speak(): void // abstract } class Dog extends Animal { speak(): void { print("Woof!") } } class Cat extends Animal { speak(): void { print("Meow!") } } // Polymorphism: same call, different results animals.forEach(a => a.speak())
06 · UML

Introduction to UML

Official Definition — OMG

UML (Unified Modeling Language) is a standard visual language dedicated to specifying, constructing, and documenting software systems.

History

UML was created in 1997 as a unification of three previously competing notations: Booch, OMT, and OOSE. It is standardized by the Object Management Group (OMG), founded in 1989.

UML 1.0 — 1997 (9 diagrams) UML 2.0 — 2005 (13 diagrams) UML 2.5.1 — Dec 2017 (14 diagrams)

UML Has Three Aspects

📖
A Language
UML has a syntax (writing rules) and standardized graphical notations, just like a programming language.
🔍
For Modeling
It abstracts system structure and behavior — it handles specification and design, not implementation.
🤝
Unified
It merges multiple prior notations into one standard, maintained by the OMG and used worldwide.

What UML Is NOT

Important Distinction

UML is a language, not a methodology. It tells you how to draw diagrams, but not how to run a software project. For methodology, you need something like UP (Unified Process).

07 · UML

UML 2.5 Diagrams

UML 2.5 provides 14 diagram types, organized into two major families: Structure diagrams (what the system IS) and Behavior diagrams (how the system EVOLVES).

The Three Modeling Perspectives

🏛️
Static (What it IS)
Class Diagram, Object Diagram — describe the structure of the system at a given moment.
⚡
Functional (What it DOES)
Use Case Diagram — describes the services the system provides to its users.
🌊
Dynamic (How it EVOLVES)
Sequence, Activity, State Diagrams — show behavior over time and interactions.
08 · UML

Using UML in Practice

Different UML diagrams are used at different phases of software development. Here's which diagram to use and when:

Analysis & Specification Phase

When gathering and documenting requirements:

DiagramPurpose
Use Case Diagram (DCU)Capture user needs and what the system must do
Sequence Diagram (DSE)Show interactions between user and system (from outside)
Activity Diagram (DAC)Represent a chain of actions / workflow

Design Phase

When designing the internal architecture of the software:

DiagramPurpose
Class Diagram (DCL)Internal structure of the software
Object DiagramInternal state at a given point in time
State-Transition DiagramHow the state of an object evolves
Sequence Diagram (DSE)Interactions inside the software between objects
Deployment DiagramPhysical components and infrastructure

The Class Diagram — Most Used

The class diagram is the heart of object-oriented design. Here's its structure:

ClassName
— Attributes —
− privateAttr : Type
+ publicAttr : Type
# protectedAttr : Type
— Methods —
+ method1() : ReturnType
− method2(param: Type)

Visibility: − private + public # protected

09 · UML

SysML vs UML

SysML (Systems Modeling Language) is a language derived from UML, designed for broader systems engineering — not just software.

🔧
Derived from UML
SysML reuses 7 UML diagrams and adds 2 new ones specific to systems engineering.
🏭
Broader Scope
Targets mechanical, electronic, and software components — not just code. Uses "blocks" instead of classes.
✂️
Lighter Syntax
Removes some UML constructs that are irrelevant to hardware/systems, making it simpler to use in engineering contexts.

Key Differences

FeatureUMLSysML
Primary domainSoftware systemsAny engineered system
Core building blockClassBlock
Number of diagrams149 (7 from UML + 2 new)
Requirements modelingLimited (via use cases)Native Requirements Diagram
Parametric modelingNot supportedParametric Diagram
When to Use Which

Use UML for pure software projects. Use SysML when your system involves hardware, embedded systems, robotics, or any mix of physical and digital components.

10 · Process

UML vs Unified Process (UP)

Critical Distinction

UML is NOT a methodology. It's a notation. The Unified Process (UP) is the methodology that tells you how to develop software, while UML provides the language to describe it.

The 4 Phases of UP

🚀
1. Inception
Launch the project, identify risks, assess feasibility. Define the project scope and make the go/no-go decision.
📐
2. Elaboration
Specify requirements, define the architecture, and set quality targets. The most critical phase.
🔨
3. Construction
Build the first complete version of the software through iterative development sprints.
🚢
4. Transition
Deploy to users, test in production, fix bugs, configure, and train users.

UP Characteristics

Iterative Incremental Generic (adaptable)

Because UP is generic, it has many specific variants adapted for different contexts:

RUP — Rational UP XUP — eXtreme UP 2TUP — 2-Track UP EUP — Enterprise UP
Summary

UML = the pencil. UP = the architectural process. You need both: a language to express your designs, and a process to guide your development.
