01

What is a Use Case Diagram?

Before any development begins, you need to answer one fundamental question:

💡 The central question "What will the software be used for?" — A Use Case Diagram (UCD) answers this question precisely.

A Use Case Diagram (UCD) is a UML diagram that represents the functions of a system from the user's point of view. It is one of the most important diagrams in UML because it captures what the system does, not how it does it.

🎯

Define Main Uses

Identify and document the primary functionalities the system must provide.

🌐

Define Environment

Show who (or what) interacts with the system from the outside.

⬜

Define Boundaries

Clearly mark what is inside the system and what is outside of it.

A UCD is made of three core elements:

// Formula: UCD = Actors + Use Cases + Relations
02

Actors

An actor represents a coherent set of roles played outside the system by a person, process, or entity that interacts with the system (to use or maintain it).

Key insight An actor is not necessarily a human being. It can be another system, a timer/scheduler, or any external entity that triggers behavior in your system.
Type Role Example
Primary Uses the main functions of the system. Triggers use cases. Direct beneficiary. Customer at an ATM; Customer placing an order on e-commerce
Secondary Performs maintenance or support tasks. May be called during execution of a use case. Not a direct beneficiary. Cash replenisher at ATM; Bank system verifying account

Actors are shown as stick figures (human actors) or labeled rectangles with «actor» stereotype (system actors). The name of an actor should reflect a role, not a person's name.

03

Use Cases

A use case describes a visible functionality offered by the system to its actors. Each use case represents one complete interaction between an actor and the system.

🔵

Primary Use Case

Directly triggered by an actor. Visible from outside the system. Named with an infinitive verb (e.g., "Place Order", "Withdraw Money").

⚪

Internal Use Case

Not directly triggered by an actor. Linked to other use cases (often via «include»). Example: "Check Stock".

✏️ Naming convention Always name use cases with an infinitive verb phrase that describes the goal from the actor's perspective: "Place Order" ✓, "Order Placement Process" ✗

04

Relations

There are three categories of relations in a UCD. Use the tabs to explore each one:

Association (Actor ↔ Use Case) The most basic relation. It shows that an actor CAN trigger a use case. Drawn as a simple line.
Real-world reading
"A Client can Withdraw Money, Check Balance, or Deposit Cheques at the ATM."
«include» (Use Case → Use Case) Use case X always triggers use case Y as part of its execution. Y is mandatory, not optional. The arrow points from the base case to the included case.

Two common reasons to use «include»:

  • Decompose a complex use case into simpler sub-cases
  • Factor out a common behavior shared by multiple use cases (avoid duplication)
Rule to remember
With «include», the included use case is always executed — it is mandatory, not conditional.
«extend» (Use Case → Use Case) Use case X optionally extends use case Y. X is triggered only under certain conditions. The arrow points from the extending case to the base case.

Key difference from «include»: the extension is optional / conditional. You can add an extension point to mark exactly where in the base use case the extension may trigger.

Include vs Extend — Quick Rule
«include» = always happens (mandatory sub-task)
«extend» = sometimes happens (optional / conditional)
Generalization (Actor→Actor or UseCase→UseCase) A is a generalization of B if B is a special case of A. B inherits all of A's use cases but may have additional exclusive ones.
Reading this diagram
A Loyalty Member is a special kind of Customer. They can do everything a Customer can (Place Order) PLUS their own exclusive action (Use Loyalty Points).
05

How to Build a UCD

Follow these four steps in order to produce a correct Use Case Diagram:

Specify the Subject Name the system and define its boundary. Everything inside the rectangle is the system; everything outside is the environment.
Identify the Actors For each actor ask: "Who starts interactions with the system?" and "Who is notified by the system?". Distinguish primary actors (users) from secondary actors (support systems).
Identify the Use Cases Place yourself in each actor's perspective. Ask: "What goal does this actor want to achieve?" Avoid action-level details — stay at goal level. Name with infinitive verbs.
Identify the Relations Connect actors to their use cases (association). Then look for shared behaviors (include), optional extensions (extend), and hierarchies (generalization).
⚠️ Common mistakes to avoid • Don't reduce a use case to a single elementary action (too granular)
• Don't introduce temporal sequencing — UCDs don't show order
• Don't put implementation details inside use cases

Full Example — Clinic Management

Let's model a clinic system with the following requirements:

📋 System requirements Patients can request appointments online and optionally specify their doctor. The secretary confirms, modifies, or cancels appointments (always notifying the patient). Doctors manage medical records and view schedules. The head doctor edits schedules. All non-patient users must authenticate first.
